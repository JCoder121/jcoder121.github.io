The Shape of an Agent

A CS61A-Style Lecture for Builders of Turn-Based Simulators, Learning How Claude Code Works

Lecture Overview

You've written game loops before — deal a hand, wait for a player action, resolve it, check win conditions, repeat. You've written state machines and append-only logs. You know that a queue is just "things I still owe processing to." This lecture asks you to look at Claude Code through that exact lens, because that's all it is: an append-only array of messages, and a while-loop that keeps a model and a set of functions talking to each other until the model stops asking for help.

There is no planner. No graph. No hidden reasoning engine bolted on top. The "intelligence" is entirely inside the model; everything you build in this course is the harness that gives that model hands — the ability to read files, write files, and run shell commands on your actual machine, instead of just producing text nobody acts on.

1. What Is an Agent?

Concept

Strip away the marketing and an "agent" is two things wired together:

  1. A model that can ask for things. Instead of only returning text, the model can return a request shaped like a function call: "call read_file with path: "foo.js"." It doesn't know what read_file does — it only knows the name and the shape of the arguments, from a schema you gave it. This is your familiar interface: a name, a signature, and a contract, with no implementation attached.
  2. A harness that actually runs things. Something on the your side of the wire reads that request, executes the real function (opens the real file, on the real disk), and reports back what happened. The model never touches your filesystem directly — it only ever sees text you choose to show it.

Loop those two together — model asks, harness runs, harness reports back, model asks again — and you have an agent. That loop is the entire "trick." Everything else in this course (tool schemas, the permission gate, error handling) is detail around that loop, not a replacement for it.

Put in game terms, since you've built these before: it's a turn-based match between two players, model and harness. The model's move is "I'd like to call these tools." The harness's move is "here's what happened when I did." Neither player can skip their turn arbitrarily — the model can't touch the disk, and the harness can't invent what the model would say next. Play continues until the model makes a move with no tool calls in it — that's resignation, i.e. "I'm done, here's my final answer."

Why It Matters

If you've only used Claude Code as an end user, its behavior can feel like it's "thinking" in some deep, opaque way about your codebase — planning multi-step refactors, deciding what to check next. It isn't planning in any special sense. It's calling a model repeatedly, feeding back whatever the model most recently asked to see. Every "decision" you observe (read this file, then that one, then edit this line) is the model choosing its next move given everything that's happened so far — the same way a chess engine's next move depends on the board state, not on some separate planning subsystem bolted outside the loop.

Understanding this precisely is what lets you debug a real agent later: if it does something wrong, the bug is in one of exactly three places — the tool's description (the model misunderstood the contract), the tool's implementation (runTool did the wrong thing), or the loop (results weren't fed back correctly). There is no fourth place to look.

2. The Message Protocol as a Data Structure

Concept

Everything the model has ever seen or said lives in one value: messages, a Anthropic.MessageParam[] — an array. Not a tree, not a graph with branches — a flat, append-only array, exactly like a log file or an event-sourced history you'd design for a game replay. Every turn appends one or more entries. Nothing is ever edited or removed in place (lecture 06's crash recovery pops a bad entry off the end — even the one exception is a stack operation, not a mutation).

Each entry (MessageParam) has two fields:

{ role: "user" | "assistant", content: string | ContentBlockParam[] }

(the | means "either/or" — e.g. role is either "user" or "assistant", and content is either a plain string or an array of blocks)

Only two roles exist. "user" means "the harness said this" (your typed input, or a tool's result — more on that below). "assistant" means "the model said this." There's no third role for "the tool" or "the system" mid-conversation — the system prompt is passed separately, once, outside this array entirely (you'll see SYSTEM_PROMPT passed as its own argument in callModel, not woven into messages).

content is usually not a plain string once tools are involved — it's an array of content blocks, and a single turn can mix several:

  • { type: "text", text: "..." } — the model talking to the human.
  • { type: "tool_use", id: "toolu_01A...", name: "read_file", input: { path: "foo.js" } } — the model asking the harness to run something. Note it has an id — a unique tag for this specific request, generated by the model/API, not by you.
  • { type: "tool_result", tool_use_id: "toolu_01A...", content: "...", is_error: false } — the harness's answer to one specific tool_use, tagged by matching tool_use_id back to the request's id.

That id / tool_use_id pairing is not cosmetic. If the model asks for three tools in one turn, you owe it three separate tool_result blocks, each carrying the matching tool_use_id — otherwise the model has no way to know which result answers which question, the same way you'd need a request ID to match async responses to their callers in any request/response system.

Why It Matters

This one fact — messages is a flat array, tool results ride back in as a "user" message — resolves almost every "wait, why is it shaped like that" moment you'll have while implementing agentLoop in lecture 05. It's easy to instinctively want a "tool" role, or to want to mutate a previous assistant message once its tool call resolves. Both instincts are wrong for this API, and both would break the model's ability to reconstruct what happened, because the model re-reads the entire array from scratch on every single call — it has no memory between calls except what's in messages. If you silently dropped or edited an entry, the model would be reasoning from a version of history that never happened.

This also explains why the harness, not the model, decides when the conversation is "done": the loop's termination condition is purely structural — does the most recent assistant turn contain any tool_use blocks? — not a signal the model has to explicitly emit.

Your Task

Predict, don't write code: given the two-content-block example above (a tool_use for read_file sitting alongside text), if the harness runs that tool and gets back the string "function foo() {...}", what does the next entry appended to messages look like — role and content shape, not full JSON?

Checkpoint

It's { role: "user", content: [{ type: "tool_result", tool_use_id: "toolu_01A...", content: "function foo() {...}", is_error: false }] }. If your answer said role: "tool" or put the result directly as a bare string for content, re-read the block shapes above — this is the single most common first-guess mistake, and section 6's unlock questions will test it again.

Common mistake: conflating "the tool ran" with "the model knows the tool ran." The model only knows what's in messages the next time you call it. Running runTool() and never pushing its result is a silent bug — the model will just repeat the same request forever, because as far as it can tell, it never got an answer.

3. A Full Trace: Watching Nano Fix a Bug

Concept

This is the centerpiece of the lecture. Below is a complete, real conversation — the kind agentLoop will drive once you've built it — annotated line by line. The scenario: fixtures/convert.js (a file already sitting in this repo, which you'll use for real in lecture 06) has a bug — Celsius-to-Fahrenheit is computed with the fraction flipped:

function celsiusToFahrenheit(c) {
  return (c * 5) / 9 + 32; // BUG: should be (c * 9) / 5 + 32
}

We type: "There's a bug in fixtures/convert.js — the temperature conversion looks wrong. Can you find and fix it?"

Here is the entire messages array's evolution, one push at a time. callModel(messages) is called three times total; each call sees the entire array built so far.

messages = []                                         // REPL starts, empty log

──── We type our request; REPL pushes it ────
messages[0] = {
  role: "user",
  content: "There's a bug in fixtures/convert.js — the temperature
             conversion looks wrong. Can you find and fix it?"
}

──── callModel(messages) call #1 — model sees just [0] ────
response.content = [
  { type: "text", text: "I'll take a look." },
  { type: "tool_use", id: "toolu_01A", name: "read_file",
    input: { path: "fixtures/convert.js" } }
]
messages[1] = { role: "assistant", content: response.content }  // pushed VERBATIM

──── Harness executes the tool_use from [1] ────
runTool("read_file", { path: "fixtures/convert.js" })
  → "function celsiusToFahrenheit(c) {\n  return (c * 5) / 9 + 32; ...}"
  (read_file is NOT gated — no y/n needed, it can't change anything)

messages[2] = {
  role: "user",
  content: [{ type: "tool_result", tool_use_id: "toolu_01A",
              content: "function celsiusToFahrenheit(c) {...}",
              is_error: false }]
}

──── callModel(messages) call #2 — model sees [0, 1, 2] ────
response.content = [
  { type: "text", text: "Found it — the formula multiplies by 5/9
                          instead of 9/5. Fixing." },
  { type: "tool_use", id: "toolu_01B", name: "edit_file",
    input: { path: "fixtures/convert.js",
              old_string: "return (c * 5) / 9 + 32;",
              new_string: "return (c * 9) / 5 + 32;" } }
]
messages[3] = { role: "assistant", content: response.content }

──── Harness executes the tool_use from [3] ────
edit_file IS gated (lecture 04) — confirm() prints a preview and waits.
We type "y". runTool(...) applies the edit, returns "Edited fixtures/convert.js"

messages[4] = {
  role: "user",
  content: [{ type: "tool_result", tool_use_id: "toolu_01B",
              content: "Edited fixtures/convert.js", is_error: false }]
}

──── callModel(messages) call #3 — model sees [0, 1, 2, 3, 4] ────
response.content = [
  { type: "text", text: "Fixed — celsiusToFahrenheit now multiplies
                          by 9/5 before adding 32." }
]
messages[5] = { role: "assistant", content: response.content }

──── No tool_use blocks in [5] → loop returns. Control back to REPL. ────

Six entries. Three round trips to the model. One human "y". Nothing about this is more complicated than that — every real Claude Code session, however long, is this same pattern run many more times.

Why It Matters

Notice what the model never sees: your filesystem, the exit code convention of edit_file, whether the y/n prompt happened. It only ever sees the six entries above, as text. Every capability the model appears to have — "it read the file," "it understood the bug," "it fixed it" — is entirely mediated by what the harness chooses to put into messages[2] and messages[4]. Get the content of a tool_result wrong (truncate it, garble it, forget to include the error) and the model's next move gets worse, with no other cause. This is the leverage point you'll be implementing across lectures 02–05: the loop's shape is fixed and small; almost all of the engineering is what string does runTool return.

  1. How many entries are in messages at the exact moment callModel is invoked for the second time (right before that call, not after)?
  2. messages[2] and messages[4] both have role: "user" — but neither one is something a human typed. What is the rule for which role a message gets: is it "who is speaking," or something else?
  3. messages[1].content[1].id is "toolu_01A". What later entry references that exact string, and under what field name?
  4. Suppose runTool("edit_file", ...) had thrown an error instead of succeeding (say, old_string matched zero times). Based on what you know about tool_result blocks having an is_error field, what would messages[4] most plausibly contain instead, and would the loop stop or continue?
  5. messages[1] contains both a text block and a tool_use block in the same array. Is that a special case, or does it follow directly from content being defined as ContentBlockParam[]?

5. Environment Setup

Concept

Two Node/TS-isms worth naming before you touch a terminal:

  • tsx is a tool that runs .ts files directly — npx tsx index.ts — without a separate compile step. Node itself can't execute TypeScript's type annotations natively; tsx strips/transpiles on the fly, the same way python3 foo.py just runs Python without a "build" phase. You will not touch tsc or any bundler in this course — that's explicitly out of scope (see "What we skipped" below).

6. Problem 0: Fix the Warm-up (1 pt)

Concept

Near the top of index.ts:

// A tiny warm-up used by lecture 01. Fixing this trivial value is Problem 0 —
// the "does my environment run at all?" check, 61A's `python3 ok -q 00` moment.
export function warmup(): number {
  return 61 * 1000 - 1; // Problem 0: something is off by two...
}

export function warmup(): number is your first TypeScript-ism worth naming explicitly: export makes the function importable from other files (that's how tests/00_setup.test.ts gets at it — import { warmup } from "../index.ts"), and : number is a return-type annotation — a promise to the compiler that this function always hands back a number, checked at compile time rather than discovered at runtime like you'd get away with in plain JS.

The bug: 61 * 1000 - 1 evaluates to 60999. The test wants 61001 — 61 * 1000 + 1. The comment even tells you it's "off by two": 60999 to 61001 is a swing of 2, caused by one flipped operator.

Why It Matters

This problem carries zero conceptual weight on its own — it's not testing whether you understand agents. It's testing whether your toolchain works end to end: can you edit index.ts, save it, and have ./ok 00 pick up the change and go green? Every one of the next five lectures assumes yes. This is the "hello world" of the course, and — like a real hello world — the value isn't the code, it's confirming nothing between you and the test runner is broken.

Key Takeaways

  1. An agent is a model that can ask, plus a harness that runs and reports back. The model never touches your machine directly. Every capability you'll ever see it exhibit is mediated by what your runTool returns as a string.
  1. messages is one flat, append-only array — never a tree, never mutated in place. The model has no memory across calls except what's in this array; it re-reads the whole thing every single time.
  1. Only two roles exist, and "user" doesn't mean "human." It means "the harness said this" — which covers both what you typed and every tool_result, because tool results are the harness's turn, just like your typed input is.
  1. tool_use.id and tool_result.tool_use_id must match. This is the invariant that lets a model ask for several tools in one turn and still know which answer belongs to which question — the same discipline as matching async request IDs to their callers.
  1. The loop's termination condition is structural, not semantic: no tool_use blocks in the most recent assistant turn means "done." The model never emits an explicit "I'm finished" signal — you infer it from the shape of its last move.
  1. This course builds one file, five sections, in dependency order. TOOLS and GATE before MODEL; MODEL before LOOP; LOOP before REPL. Each lecture fills exactly one section, using the exact identifiers already declared in the skeleton.

What we skipped (for future study or future lectures)

  • Streaming responses (the loop here always waits for a complete Message before acting).
  • What actually happens inside preview / confirm — that's lecture 04; today you only saw that a gate exists in the trace, not how it's implemented.
  • The internals of runTool for each of the 5 tools — lectures 02–03.
  • Anything about tsc, bundlers, or TypeScript build tooling — tsx running files directly is all this course uses.
  • Multi-step planning, subagents, context compaction — real Claude Code has all of these; this course's whole point is that the core loop needs none of them to work.

Snippet 1: The two-part definition

An agent = (1) a model that can emit structured requests (tool_use blocks) instead of only prose, given a schema describing what it's allowed to ask for, plus (2) a harness that reads those requests and decides, on its own authority, whether and how to execute them. The model producing a tool_use block is inert text until the harness's loop (agentLoop, lecture 05) chooses to call runTool. For gated tools (bash, write_file, edit_file — lecture 04's GATED set), that choice runs through a human y/n first. Nothing happens to your filesystem merely because the model "wants" it to.

Snippet 2: Message shapes 
// A plain user turn (what the REPL pushes for your typed input):
{ role: "user", content: "fix the bug in foo.js" }

// An assistant turn with mixed content — pushed VERBATIM from response.content:
{
  role: "assistant",
  content: [
    { type: "text", text: "I'll check that file." },
    { type: "tool_use", id: "toolu_01A", name: "read_file", input: { path: "foo.js" } },
  ],
}

// The harness's answer to that tool_use — note role is "user", not "tool":
{
  role: "user",
  content: [
    { type: "tool_result", tool_use_id: "toolu_01A", content: "...file contents...", is_error: false },
  ],
}

The rule for role is purely "who owns this turn in the protocol," not "who is a human." The harness owns tool results the same way it owns your typed input — both are things the API's "assistant" (the model) did not produce.

Snippet 3: Unlock question answers
  1. 3. messages = [0] (your request), [1] (assistant + tool_use), [2] (tool_result) — three entries exist right before call #2 fires.
  2. Which "side of the wire" produced it, not "was a human typing." role: "user" means "the harness, in the broad sense" — your literal keystrokes and a tool's result are both the harness's turn to speak.
  3. messages[3].content[1].input.old_string? No — trick to watch for. "toolu_01A" is referenced by messages[2].content[0].tool_use_id, the tool_result answering that specific request. "toolu_01B" (a different id, from the edit_file call in messages[3]) is what messages[4].content[0].tool_use_id matches. Don't confuse the two ids — each tool_use gets its own.
  4. messages[4] would contain a tool_result with is_error: true and content set to the error message string (e.g. "old_string not found in fixtures/convert.js"). The loop does not stop — an error is still a result, just an unhappy one; the model sees it on call #3 and can retry with a corrected old_string. This is the self-correction behavior lecture 05's tests specifically script for.
  5. Follows directly, not a special case. content is typed as ContentBlockParam[], an array — nothing restricts it to one block. A single assistant turn commonly narrates in text and requests one or more tool_use calls in the same turn; the loop just iterates every block in response.content.
  6. The API would reject the request (a 400-style error) — Anthropic's API requires every tool_use in the prior assistant turn to have a matching tool_result before it will accept the next call. This is exactly the scenario lecture 06's crash-recovery code guards against: if a call fails mid-loop, the last assistant message is "dangling" (has unanswered tool_use blocks), so main() pops it before trying again.
