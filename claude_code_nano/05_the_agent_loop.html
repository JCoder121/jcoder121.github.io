The Agent Loop

A CS61A-Style Lecture for Builders of Turn-Based Simulators, Facing the Boss Level

Lecture Overview

Four lectures of setup end here. You have five working tools (runTool), a permission gate that pauses for a human y/n (preview + confirm), and — from lecture 01 — a complete, hand-traced mental model of what a Claude Code conversation actually looks like as data: a flat messages array, tool_use blocks the model emits, tool_result blocks the harness answers with. What you don't have yet is anything that runs. TOOLS describes tools; runTool executes them; confirm gates them — but nothing has ever called any of them in sequence, because nothing has ever talked to a model. Today you write the two functions that make all of it move: callModel, the one function that actually reaches the Anthropic API, and agentLoop, the while loop that calls the model, runs whatever it asks for, and keeps going until it stops asking.

The curriculum calls this the boss level, and it earns the name for a specific reason: it is not new difficulty so much as total composition. There is no new tool to learn, no new Node API to gloss (mostly — a couple of small ones below), no new permission concept. agentLoop is short — the reference implementation is about 30 lines — but every one of those lines calls something from a previous lecture: runTool (02–03), GATED and confirm (04), and the messages protocol you traced by hand in lecture 01 §3. If any of those four lectures left a gap in your understanding, this is where it surfaces, because agentLoop is the first code in this course that actually exercises the full chain end to end.

There's a second reason this lecture matters beyond "it's the biggest function": it's where the course's central claim gets proven, not just asserted. Lecture 01 told you an agent is "a model that can ask, plus a harness that runs and reports back," looped. That was a description of behavior you couldn't yet produce. Today you write the loop, run it against a scripted fake model, and watch your own code generate — mechanically, deterministically — the exact six-entry transcript you predicted by hand in lecture 01 §3. That's not a coincidence or a nice callback; it's the whole point of front-loading the trace before any code existed. You already know what correct output looks like. Today's job is making your code produce it.

How this lecture works: two numbered problems. Problem 5.1 is small (callModel, one API call, one new pattern). Problem 5.2 is the real work (agentLoop), broken into three lettered stages — (a) the shape of one iteration, (b) executing tool calls including the gate and error handling, (c) the termination condition — but all three stages together are checked by one target, ./ok 05, against tests/05_agent_loop.test.ts. Each stage still gets its own spec, worked example, constraints pulled directly from that test file, and unlock questions answered only in the Appendix. This lecture is worth 2.5 of the course's 10 points — the single largest point value of any lecture, matching its billing.

1. Problem 5.1: callModel — the One Function That Talks to Anthropic

Concept

Everything you've built so far — runTool, preview, confirm — runs entirely on your machine: files, shell, a human typing y/n. callModel is different in kind: it's the single function in this entire codebase that leaves your machine and talks to a remote model. Open index.ts and look at its neighbors — MODEL, SYSTEM_PROMPT, and a let client: Anthropic | undefined; declared just above it. All three exist only to support callModel; nothing else in the file touches them.

callModel makes exactly one API call, and that call needs five things to mean anything. Which model to talk to — the MODEL constant, a specific model id, not a family name. A token cap — max_tokens, a budget the reply isn't allowed to exceed, not a target it aims for. The system prompt — SYSTEM_PROMPT, the standing instructions that apply to every turn of the conversation, not just this one. The tools contract — TOOLS, the schema array lecture 02 spent an entire section teaching you to read as a contract; this is the one place in the whole project where that contract actually leaves your machine and reaches the model, over the wire, every single call — nothing about TOOLS matters to the model until this line ships it. And the history — the messages parameter you were handed, so the model sees the whole conversation, not just the latest line. Drop any one of these five and you get a worse agent in a specific way: no model to answer, no cap on runaway generation, no standing instructions, no tools it's allowed to ask for, or no memory of anything said so far.

The other thing to understand before writing this function is when the Anthropic client itself gets built. It can't happen once, at the top of the file, the moment index.ts loads — it has to happen lazily, the first time callModel actually runs. Constructing a real Anthropic client reads ANTHROPIC_API_KEY from your environment, and every test file in this course — including every test in this lecture's own suite — imports index.ts at the top, often with no API key set at all. If client construction happened at import time, simply importing the file would risk failing before a single test assertion runs, for a reason that has nothing to do with what that test is checking. Building lazily — check whether a client already exists, construct one only if it doesn't, and reuse it on every later call — defers that risk to the one code path that actually needs a key: a real call to callModel.

Why It Matters

The comment above let client says the quiet part out loud: "created lazily so importing this file never needs an API key." If new Anthropic() ran at module load time — say, as const client = new Anthropic(); at the top level — then simply import-ing index.ts (which is exactly what every test file in this course does) would try to construct a real client immediately, and depending on the SDK's behavior, that could fail or behave unexpectedly in an environment with no ANTHROPIC_API_KEY set at all — which describes every single test run in this course, including the ones you're about to write for agentLoop. Laziness — construct on first use, not on import — is what keeps ./ok 05 running "offline and free" (the ok script's own comment, lecture 01 §5) even though callModel genuinely does real network I/O when actually invoked.

Now look at agentLoop's signature in the skeleton, specifically its third parameter: model: (messages) => Promise<Anthropic.Message> = callModel. That's a default parameter — a TypeScript/JavaScript feature you haven't needed until now: if the caller doesn't supply a model argument, agentLoop uses callModel automatically. callModel is the real backend; the default value is a convenience, not the design. The design is that model can be anything shaped like callModel — and tests/05_agent_loop.test.ts never once calls the real callModel. Every single test builds a scriptedModel(responses) — a fake function returning pre-baked Anthropic.Message-shaped objects off a queue — and passes that as model instead. This is the exact same move lecture 04 taught you with Asker: confirm never called readline.createInterface directly, it depended on the narrowest interface it needed (question(prompt): Promise<string>) so tests could inject a scriptedAsker. callModel's injectability through agentLoop's model parameter is that identical seam, applied to the model instead of the human. You are about to write and fully test the boss-level function of this course without ever making a real API call — and that's not a limitation of the test suite, it's the design lesson this lecture is built to teach.

Worked example. There isn't one you can run offline — that's the point of this problem, and §1's "Why It Matters" above already explained why. The "worked example" for callModel is really: does it compile, and does it match the shape agentLoop's default parameter expects (a function from Anthropic.MessageParam[] to Promise<Anthropic.Message>)? You'll get a real, live worked example in lecture 06's acceptance run, once ANTHROPIC_API_KEY is set and main() exists to drive it.

No test in this entire course calls the real callModel. Correctness here is checked by you, by reading, and — for real, eventually — by lecture 06's live run against the actual API.

2. Problem 5.2a: One Trip Around the Loop

Concept

Before writing agentLoop in full, get the shape of a single iteration fixed in your head — you'll wrap it in while (true) in §4, but the interesting design decisions all live inside one trip around the loop, not in the looping itself.

Recall the DSA framing from the skeleton's own comment block above agentLoop: messages is an append-only array (lecture 01 §2 — never edited in place, only pushed to) and agentLoop is a while-loop draining a work queue — each iteration may discover more work (new tool_use requests) that gets queued for the next iteration by virtue of being appended to messages. One trip around the loop does exactly three things, in order:

  1. Call the model. Await the injected model parameter, passing it the current messages — not callModel directly by name, which is exactly what makes the whole function testable (§1). What comes back is an Anthropic.Message whose content field is the array of blocks (text, tool_use, or a mix of both) the model produced this turn.
  2. Push the assistant's reply, verbatim. Take that content array exactly as the API returned it — not reformatted, not filtered, not summarized — and append it to messages as one new entry playing the assistant role. Whatever mix of blocks the model produced goes in unmodified.
  3. Look at what it asked for. Walk the content blocks and find any tool_use blocks — that's next section's job.

The word verbatim in step 2 is doing real work, and it connects directly to something lecture 01 §2 already told you: the model has no memory across calls except what's in messages. If you rebuilt response.content yourself instead of pushing it as-is — say, by only keeping the text blocks and dropping the tool_use ones — the next call to model(messages) would show the model a version of its own last turn that never actually happened. It asked for read_file; your rewritten history says it didn't. The model would have no way to reconcile that, because from its perspective, everything it "remembers" is just what's sitting in messages — there's no other memory to fall back on.

Why It Matters

This is a turn-based game in the exact sense lecture 01 §1 first proposed: the model makes a move (response.content, which can hold text and one or more tool_use blocks together — lecture 01 §3's unlock question 5 already established that a turn is never restricted to one kind of block, since content is typed as a plain array), and pushing that move onto messages before you've even looked at what's in it is the harness's equivalent of writing the opponent's move down on the scoresheet before deciding how to respond to it. You don't get to silently veto or edit the model's move by rewriting history; you get to decide, in the next steps, what your own response move looks like.

Your Task

Spec. Implement the first two moves of one loop iteration inside agentLoop, in index.ts: call the injected model function with the current messages, then append one new entry to messages, playing the assistant role, whose content is exactly the response's content array — unmodified. This code lives inside the while (true) you'll add in §4; for now, focus only on getting these two moves right. The tool-execution walk (§3) and the loop's continue-or-stop decision (§4) come next, and ./ok 05 won't go green until all three stages are in place together.

Worked example. Before: messages holds one entry, { role: "user", content: "hi" }. The scripted model's response has a content array with two blocks: a text block reading "checking now", and a tool_use block requesting read_file on "a.txt". After this stage runs once: messages holds two entries — the original user entry, unchanged, plus one new assistant entry whose content is that same two-block array, verbatim. (This is exactly the scenario the prediction question below asks you to work out in full detail.)

3. Problem 5.2b: Executing Tool Calls — the Gate, the Try/Catch, the Results Array

Concept

Now the interesting part: for each tool_use block in the assistant's reply, you must produce exactly one tool_result — and which of three paths that result takes depends on what you already built in lectures 03 and 04. This section is where GATED, confirm, and runTool's throw-on-failure convention all get called from the same dozen lines, for the first time.

Walk the content blocks one at a time, in the order the model produced them, and handle each by its type:

  • A text block: print it — console.log is fine, the model's prose is meant to be read — and move on to the next block. Nothing else happens for a text block.
  • A tool_use block: first log a short one-liner naming the tool and its input, so a human watching the REPL can see what's about to happen. Then decide which of three paths this particular call takes, in order:
    • Gated, and denied. Check whether the tool's name is a member of GATED (the exact set from lecture 04, unchanged). If it is, ask for approval the same way lecture 04's confirm does, and wait for the answer. If the answer is a denial, do not call runTool at all — the whole point of the gate is that a denial means the action never happens. Instead, this block's result gets a human-readable content string containing the word "declined" (checked case-insensitively — tests/05_agent_loop.test.ts checks .toLowerCase().toContain("declined")) and an is_error flag of false. Recall lecture 04 §6's forward-reference: "a denial is a normal result, not an error" — you predicted this exact shape three lectures ago, and now you're building it for real.
    • Gated and approved, or not gated at all (read_file/list_files skip the gate question entirely): run the tool. Wrap that call in a try/catch — recall lecture 02 §5's convention, restated as a promise back then: "agentLoop will wrap every runTool call in a try/catch, and when it catches something... it converts the exception into an ordinary tool_result block with is_error: true." Today is that promise's due date.
      • It succeeds: this block's result gets whatever runTool returned as content, and is_error stays false.
      • It throws: this block's result gets the caught error's message as content (or the caught value itself, stringified, if it isn't a proper Error), and is_error becomes true.
    • Whichever of these three paths a given tool_use took, the outcome is one result object that carries the id off the request you're currently answering — the join key lecture 01 §2 called "not cosmetic," copied straight from the block you just processed. Push each one onto a local array (call it results) as you go — you're accumulating one entry per tool_use block encountered this turn, not sending anything back to messages yet.

Gloss — iterating content blocks directly. You want a loop that visits each element of the content array itself, not its indices — the same idea as list_files's .map() from lecture 02, just written as a loop keyword instead of a method call. This is the first time in the course you're writing that direct-iteration loop keyword yourself rather than reaching for an array method; look it up if the syntax for "loop over each element of an array, by value" isn't already familiar.

Why It Matters

Look at what this section does not do: it never checks "is this the kind of tool that usually throws" or special-cases bash versus read_file. Every gated tool goes through confirm the same way; every tool call goes through the same try/catch; the branch logic is uniform across all five tools, because the conventions — GATED membership, throw-on-failure — were established once, in earlier lectures, precisely so that agentLoop wouldn't need tool-specific logic here. This is the payoff of lecture 02's dispatch-table framing and lecture 04's gate: build the right primitives once, and the code that composes them stays almost embarrassingly simple. If you find yourself writing an if (block.name === "bash") anywhere in this section, stop — that's a sign you've reinvented something GATED or runTool's dispatch already solved.

The is_error/content distinction is also where you'll see the mechanism behind something that looks, from a distance, like self-correction. tests/05_agent_loop.test.ts's test (c) scripts a model that asks for read_file on a path that doesn't exist, gets back is_error: true with the real ENOENT message as content, and is scripted to reply "sorry, that file is missing" on its very next turn. Read that carefully: the reply is hardcoded into the test's script, not generated in reaction to the error — so what test (c) actually proves is narrower than "the model self-corrected." It proves the mechanism is wired correctly: the real error text genuinely reaches messages, legible and unfiltered, and the model gets a second turn to do something with it, exactly the way lecture 02 §5 promised. Whether a real model, handed that same error text, would actually produce something sensible is a claim this test can't make on its own — that's lecture 06's live run against the real API, where the reply isn't written in advance.

Hint: Short-circuit evaluation (&&, ||) lets you fold "is this tool even gated?" and "did the human approve it?" into a single boolean condition, evaluated left-to-right — the same way you'd guard an array access with arr && arr.length to avoid touching .length on something that might not exist. Only tools where the first check passes ever reach the await confirm(...) call at all.
Common mistake: Calling confirm for every tool, then checking GATED.has(...) only to decide whether to look at its answer. This means an ungated call like read_file would sit there awaiting a human's y/n for no reason — annoying in a real REPL, and a guaranteed hang against noAsker in tests (b), (c), and (e), whose Asker is deliberately built to throw new Error("gate should not have been consulted") the instant question is called. Those three tests exist specifically to catch a confirm call on a tool that was never supposed to trigger the gate.

This code sits between the assistant-push you wrote in §2 and the return-or-continue decision you'll write in §4.

Running this stage's logic produces a results array with one entry: content containing the command's stdout and its exit code, is_error false, and an id matching the tool_use block's id. Now suppose instead the human answers "n" when asked to approve a real (gated) bash call — the results array still has exactly one entry, but its content contains the word "declined," is_error is still false, and — critically — the command never ran. 4. Problem 5.2c: Feeding Results Back and Knowing When to Stop

Concept

You now have a results array — zero or more tool_result objects, one per tool_use block the model's last turn contained. Two things remain, and they're really one decision with two branches:

If results is empty — the model's last turn had no tool_use blocks at all, meaning it just talked (recall lecture 01 §2's "Why It Matters": "the loop's termination condition is purely structural... does the most recent assistant turn contain any tool_use blocks?") — agentLoop is done: exit the function immediately, with no value to return. Control goes back to whatever called agentLoop (eventually main(), lecture 06).

If results has at least one entry — send them all back in a single new message, not one message per result: one entry playing the user role (the API's shape — the assistant asks, the harness/"user" answers), whose content is the entire results array. Then loop again, back to the top of §2's step 1 — call the model, with the now-longer messages.

That "single message" detail matters and is easy to get subtly wrong: if the model's last turn asked for three tools in one go, you send back one new message whose content holds all three results together — not three separate messages, one per result. This mirrors §2's "push the assistant's whole reply verbatim as one message" exactly, just on the harness's side of the exchange instead of the model's: one turn, one message, possibly many blocks inside it.

Why It Matters

This is the exact rhythm lecture 04 §2's unlock question previewed before you'd written a line of agentLoop: "every function on the call path... ends up async... for the same one reason repeated at each link." Here's where that chain actually closes the loop — literally. agentLoop is async because it awaits model(...) and (inside §3) awaits confirm(...); the while (true) around all of it is what turns "one exchange" into "a whole conversation," continuing exactly as long as the model keeps asking for tools and stopping the instant it doesn't.

It's also the reason this function mutates its caller's array instead of returning a new one. Look at agentLoop's return type again: Promise<void> — not Promise<Anthropic.MessageParam[]>. It never returns the updated conversation; it mutates the messages array it was handed, via .push(...), in place. That's deliberate, and it's not just an implementation detail — lecture 06's REPL depends on exactly this behavior: main() will keep one messages array alive across multiple calls to agentLoop (one per line you type at the prompt), and it only works because each call appends to the same array rather than handing back a fresh one main() would have to remember to reassign. If agentLoop built and returned a new array instead of mutating the one it was given, the REPL's persistence across turns would require an extra assignment at every call site — a small thing, but not what the skeleton's signature (Promise<void>, no return value) is set up for.

Hint: The check that decides "loop again or return" is a single condition on results.length — nothing fancier (no separate boolean flag tracked across the for...of from §3, no counting text blocks). Whatever you push onto messages for the "loop again" branch is one message, not a for loop that pushes once per entry in results.
Common mistake: Pushing the results array's entries onto messages one at a time (for (const r of results) messages.push({ role: "user", content: [r] })) instead of once, as a single message whose content is the whole array. Watch closely here: every scripted response in tests/05_agent_loop.test.ts happens to contain exactly one tool_use block per turn — so with only one result to push either way, this bug produces identical output to the correct version, and ./ok 05 would not catch it. This is exactly the kind of gap lecture 03 §2 warned you about with write_file's untested return value: passing every test in the file isn't the same claim as being correct for every input the model could send. A model that asks for two tools in one turn (which real Claude Code does routinely) would expose the bug immediately; nothing in this lecture's suite happens to ask for that.

Key Takeaways

  1. callModel is the one function that leaves your machine, and it's built to be lazy — the client is constructed only on first use, via the assignment form of ??, not at module load — so that importing index.ts — which every test file does — never requires a real API key. No test in this course calls it directly; that absence is itself the lesson.
  1. agentLoop's model parameter is an injectable seam, the same pattern as lecture 04's Asker: a default value (= callModel) for real use, overridable by a scripted fake for tests. This is what makes the boss-level function fully testable offline.
  1. One trip around the loop is exactly three moves: call the model; push its reply verbatim (never edited, never filtered — the model must see its own requests, unmodified, on the next call); walk the content blocks and build one tool_result per tool_use.
  1. Every tool_use resolves through the same three-way branch: gated-and-denied → a "declined" result with is_error: false; ungated-or-approved-then-thrown → the caught error's message with is_error: true; ungated-or-approved-then-succeeded → the return value with is_error: false. No tool gets special-cased — the branch is uniform because GATED and the throw-on-failure convention were built once, in earlier lectures, specifically so agentLoop wouldn't need to know the difference between bash and read_file.
  1. Tool results ride back as one new message playing the user role, whose content is the entire results array — not one message per result — the mirror image of pushing the assistant's whole turn as one message. The loop terminates the instant a turn produces zero tool_use blocks; that check is structural, not something the model explicitly signals.
  1. agentLoop mutates its caller's messages array rather than returning a new one (Promise<void>, not Promise<MessageParam[]>) — a design choice lecture 06's REPL depends on: one long-lived array, persisted across many calls to agentLoop, one call per line typed at the prompt.
  1. You predicted this exact output four lectures ago. Lecture 01 §3's trace wasn't a warm-up exercise disconnected from the rest of the course — it was the specification agentLoop had to satisfy, written down before any of the code existed to satisfy it.

What we skipped (for future study or future lectures)

  • main() and the readline REPL that actually drives agentLoop from a real terminal, plus crash recovery (popping a dangling assistant message) — lecture 06, in full.
  • Streaming responses — agentLoop always waits for one complete Anthropic.Message before acting; a streaming version would need to handle partial content blocks arriving incrementally, a meaningfully different shape than this course covers (curriculum: explicitly out of scope).
  • Anything about what happens if model(messages) itself throws (a network error, a malformed response) — not exercised by this lecture's tests, and not handled specially inside agentLoop itself; lecture 06's main() is where a top-level try/catch around the whole REPL loop catches this class of failure.
  • Multiple tool calls executed concurrently instead of one after another — this course's loop processes tool_use blocks sequentially within a turn (a plain for...of), which is simpler and sufficient for everything this course tests, though a production agent might parallelize independent reads.
  • Token/context limits, conversation truncation, or summarizing old turns to keep messages from growing unbounded — real Claude Code has to deal with this; this course's toy conversations never get long enough to need it (curriculum: "context compaction... v2 territory").

Snippet 4: Full agentLoop 
export async function agentLoop(
  messages: Anthropic.MessageParam[],
  ask: Asker,
  model: (messages: Anthropic.MessageParam[]) => Promise<Anthropic.Message> = callModel,
): Promise<void> {
  while (true) {
    const response = await model(messages);

    // The assistant's turn joins the history VERBATIM — including its
    // tool_use blocks. The model must see its own requests next turn.
    messages.push({ role: "assistant", content: response.content });

    const results: Anthropic.ToolResultBlockParam[] = [];
    for (const block of response.content) {
      if (block.type === "text") console.log(`\n${block.text}`);
      if (block.type !== "tool_use") continue;

      console.log(`● ${block.name}(${JSON.stringify(block.input).slice(0, 120)})`);

      let content: string;
      let is_error = false;
      if (GATED.has(block.name) && !(await confirm(ask, block.name, block.input))) {
        // A denial is just information, sent back like any other result.
        content = "The user declined this action. Ask before retrying, or try another approach.";
      } else {
        try {
          content = runTool(block.name, block.input);
        } catch (err: any) {
          // Errors go BACK TO THE MODEL — watch it read them and self-correct.
          content = String(err?.message ?? err);
          is_error = true;
        }
      }
      results.push({ type: "tool_result", tool_use_id: block.id, content, is_error });
    }

    // No tool requests → the model is done. Hand control back to the REPL.
    if (results.length === 0) return;

    // Tool results are sent as a *user* message — that's the API's shape:
    // assistant asks, "user" (us, the harness) answers.
    messages.push({ role: "user", content: results });
  }
}
Snippet 4: Full agentLoop — unlock answers
  1. Six entries, roles in order: ["user", "assistant", "user", "assistant", "user", "assistant"]. Walking it: messages[0] is the original "explore" prompt; call #1 returns the list_files tool_use, pushed as messages[1] (assistant); its result is pushed as messages[2] (user); call #2 returns the read_file tool_use, pushed as messages[3] (assistant); its result is pushed as messages[4] (user); call #3 returns plain text with no tool calls, pushed as messages[5] (assistant), and since that turn produced zero tool_use blocks, results.length === 0 and the loop returns without pushing a seventh entry. This is precisely tests/05_agent_loop.test.ts test (e).
  2. The shape stays identical either way — one tool_use in, one tool_result out, regardless of whether that result represents an approval-and-success or a denial. A denied bash call still produces exactly one tool_result entry in results, still gets pushed inside exactly one { role: "user", content: results } message, same as an approved-and-successful one. Only the content string and the fact that is_error stays false either way (a denial isn't an error — lecture 04 §6) differ; the message-count shape is invariant to which of the three branches (denied / succeeded / threw) a given call took.
  3. No, it would not pass — or at least not for the reason the test author intended. tests/05_agent_loop.test.ts's assertions (e.g. expect(messages).toHaveLength(2), expect(messages.map((m) => m.role)).toEqual([...])) all inspect the same array reference the test itself created and passed into agentLoop as the first positional argument, then check it after await agentLoop(...) has resolved — with no reassignment on the test's side. If agentLoop's body reassigned its local messages parameter to a new array via .concat(...) instead of mutating in place via .push(...), that reassignment only rebinds the local variable inside agentLoop's own scope; the original array the test is holding a reference to never gets any new entries appended to it at all. The test would see the original, un-grown array — likely still just the single initial entry — and every length/role assertion would fail.
  4. The work queue is "drained" exactly when a turn's results array is empty, i.e. the model's most recent turn contained zero tool_use blocks — checked after that turn's assistant message has already been pushed onto messages (§2's step 2 always runs before this check), but before anything would be pushed for a "next round" that, in this case, never happens. This is the same structural (not semantic) termination rule lecture 01 §2 first stated: the harness infers "done" from the shape of the last turn, not from any explicit signal the model has to emit.
